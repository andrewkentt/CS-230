# CS-230

For this project, the client was The Gaming Room, and they wanted to expand their existing game, Draw It or Lose It, into a web-based application that could run across multiple platforms instead of just Android. The main goal was to design a system that could support multiple users at the same time while still being scalable, secure, and consistent. This meant focusing on things like server-side architecture, cross-platform compatibility, memory and storage management, and overall system design.

One thing I think I did well in this project was clearly organizing the design document and explaining my decisions in a way that made sense. I focused on keeping everything consistent and making sure each section connected back to the client’s requirements. Working through the design document before jumping into code was actually really helpful because it forced me to think through the structure of the system ahead of time instead of just figuring things out as I went.

If I had to go back and revise one part, I would probably improve the level of detail in some of the earlier sections, especially when comparing platforms. At the time, I focused more on getting the main ideas down, but I think I could have added a little more depth or examples to strengthen those sections. It would make the document feel more complete and give better justification for my decisions.

When it came to understanding the user’s needs, I focused on what the client was asking for, like supporting multiple players, running on different platforms, and keeping the system efficient. Designing with the user in mind is important because if the system doesn’t meet their needs, then even a well-built application won’t be useful. Making sure the design stays aligned with those requirements helps guide decisions throughout the process.

Overall, my approach to designing the software was to break the problem down into smaller pieces and handle each part step by step. I used object-oriented design concepts and patterns like Singleton and Iterator to help manage the system more effectively. In the future, I would continue using this kind of structured approach, but I would also spend more time planning and possibly creating more detailed diagrams or prototypes earlier on. That would make it easier to visualize the system and catch potential issues before development begins.
